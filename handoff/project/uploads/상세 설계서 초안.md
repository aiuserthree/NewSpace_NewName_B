
# 공간 이름 공모전 — 페이지 단위 상세 설계서 (기능정의서)

> 본 문서는 「공간 이름 공모전 웹페이지 기획서(초안)」을 기반으로,
> 각 화면(메뉴) 단위의 **상세 기능정의 + 비즈니스 로직 시나리오**를 정리한 문서입니다.
> 대상 화면: 인트로 → 신청자 정보 입력 → 이름 제출 → 제출 완료 → 내용 조회

---

## 0. 공통 정의 (Global Spec)

### 0-1. 화면(페이지) 목록 및 코드
| 페이지 코드 | 화면명 | 역할 | 진입 경로 |
|------------|--------|------|-----------|
| P0 | 인트로 / 랜딩 | 공모전 소개, 참여 유도 | 카카오톡 링크 클릭 |
| P1 | 신청자 정보 입력 | 이름·연락처 입력, 중복 판별 | P0 [참여 시작] |
| P2 | 이름 아이디어 제출 | 3개 공간 이름 입력·제출 | P1 (신규 판정) |
| P3 | 제출 완료 | 완료 안내, 2차 투표 안내 | P2 [제출하기] |
| P4 | 내 제출 내용 조회 | 제출 내역 읽기 전용 | P1 (기존 판정) [내용 확인하기] |

### 0-2. 공통 데이터 항목 (필드 정의)
| 필드명 | 코드 | 타입 | 필수 | 제약/비고 |
|--------|------|------|------|-----------|
| 신청자 이름 | applicant_name | String | Y | 1~20자, 공백만 입력 불가 |
| 핸드폰 번호 | phone_number | String | Y | 숫자만, 하이픈 제거 저장, 고유 키(Unique) |
| 본당 이름1(필수) | main_hall_name1 | String | Y | 1~30자 |
| 본당 이름2(선택) | main_hall_name2 | String | N | 1~30자 |
| 소예배실 이름1(필수) | chapel_name1 | String | Y | 1~30자 |
| 소예배실 이름2(선택) | chapel_name2 | String | N | 1~30자 |
| 새가족부실 이름1(필수) | newfamily_name1 | String | Y | 1~30자 |
| 새가족부실 이름2(선택) | newfamily_name2 | String | N | 1~30자 |
| 제출 일시 | submitted_at | DateTime | Y(자동) | 서버 기준 자동 기록 |

### 0-3. 공통 정책
- 핸드폰 번호를 **중복 방지 고유 키**로 사용한다.
- 1인당 각 공간별 **필수 1개 + 선택 1개 = 공간당 최대 2개**를 제출한다.
- **예시 이름은 제공하지 않는다.** (창의성 저해 방지)
- 제출 = **최종 완료**. (수정 기능 미구현 / 조회만 가능)
- 세 공간 이름은 **한 화면(P2)에서 한 번에** 제출한다.
- 흐름 원칙: 다중 페이지 '다음' 반복 방식 배제 → **스크롤 기반 단일 페이지 + 최소 단계**로 이탈 방지.

### 0-4. 공통 예외/에러 처리 원칙
| 상황 | 처리 |
|------|------|
| 필수값 미입력 | 해당 필드 하단 경고 노출, 다음 단계 차단 |
| 번호 형식 오류 | "올바른 핸드폰 번호를 입력해 주세요." |
| 네트워크/서버 오류 | "일시적인 오류가 발생했습니다. 잠시 후 다시 시도해 주세요." |
| 중복 제출 시도(동일 번호) | P4(조회)로 유도, 신규 제출 차단 |

---

## 1. P0 — 인트로 / 랜딩 페이지

### 1-1. 화면 목적
공모전의 취지와 참여 방법을 안내하고, 참여를 시작하도록 유도한다.

### 1-2. 화면 구성 요소
| 구분 | 요소 | 내용/설명 |
|------|------|-----------|
| 헤드라인 | 메인 타이틀 | "우리 공간의 이름을 함께 지어주세요" |
| 소개 | 공모 취지 | 신축·리모델링 공간 이름을 교인 참여로 공모 |
| 대상 안내 | 대상 공간 요약 | ① 본당 ② 소예배실 ③ 새가족부실 |
| 참여 안내 | 참여 방법 | 정보 입력 → 이름 제출 (2단계) 안내 |
| 시각 요소 | 대표 이미지 | 조감도/사진 중심의 시각 강조 배치 |
| CTA | [참여 시작] 버튼 | 클릭 시 P1로 이동 |

### 1-3. 기능 정의
| 기능 ID | 기능명 | 설명 |
|---------|--------|------|
| F0-01 | 콘텐츠 노출 | 취지·참여방법·대상공간 정적 노출 |
| F0-02 | 참여 시작 이동 | [참여 시작] 클릭 → P1 이동 |

### 1-4. 비즈니스 로직 시나리오
    [시나리오 P0-S1] 정상 진입
    1. 사용자가 카카오톡 채널로 받은 링크를 클릭한다.
    2. 시스템은 P0(인트로)를 표시한다.
    3. 사용자는 공모 취지·대상 공간·참여 방법을 확인한다.
    4. 사용자가 [참여 시작]을 누른다.
    5. 시스템은 P1(신청자 정보 입력)으로 이동한다.

---

## 2. P1 — 신청자 정보 입력 페이지

### 2-1. 화면 목적
신청자의 이름·핸드폰 번호를 입력받고, 번호 기준으로 신규/기존 여부를 판별한다.

### 2-2. 화면 구성 요소
| 구분 | 요소 | 내용/설명 |
|------|------|-----------|
| 입력 | 이름 | 텍스트 입력 (필수) |
| 입력 | 핸드폰 번호 | 숫자 입력 (필수, 중복 확인 키) |
| 안내 | 개인정보 안내 문구 | (열린 이슈 — 확정 시 노출) |
| CTA(신규) | [다음] | 중복 없을 때 P2 이동 |
| CTA(기존) | [내용 확인하기] | 중복 감지 시 P4 이동 |
| 안내(기존) | 중복 메시지 | "이미 신청하신 분입니다." |

### 2-3. 기능 정의
| 기능 ID | 기능명 | 설명 |
|---------|--------|------|
| F1-01 | 입력값 검증 | 이름/번호 필수 및 형식 검증 |
| F1-02 | 번호 정규화 | 하이픈·공백 제거 후 숫자만 저장 |
| F1-03 | 중복 조회 | 입력 번호가 기존 제출 데이터에 존재하는지 확인 |
| F1-04 | 분기 처리 | 신규 → P2 / 기존 → 중복 안내 + [내용 확인하기] |

### 2-4. 비즈니스 로직 시나리오
    [시나리오 P1-S1] 신규 신청자
    1. 사용자가 이름과 핸드폰 번호를 입력한다.
    2. 사용자가 [다음]을 누른다.
    3. 시스템은 필수값·형식을 검증한다.
       3-1. 미입력/형식 오류 시 → 경고 노출, 진행 차단.
    4. 시스템은 입력 번호를 정규화(하이픈 제거)한다.
    5. 시스템은 해당 번호의 기존 제출 여부를 조회한다.
    6. 기존 데이터가 없으면(신규) → 입력 정보를 임시 보관하고 P2로 이동한다.

    [시나리오 P1-S2] 기존 신청자(동일 번호 재접근)
    1. 사용자가 이름과 이미 제출한 핸드폰 번호를 입력한다.
    2. 사용자가 [다음]을 누른다.
    3. 시스템은 번호 정규화 후 중복 조회를 수행한다.
    4. 기존 데이터가 존재하면 →
       4-1. "이미 신청하신 분입니다." 메시지를 노출한다.
       4-2. [내용 확인하기] 버튼을 노출한다.
    5. 사용자가 [내용 확인하기]를 누르면 P4(조회)로 이동한다.
    6. 신규 제출(P2) 진행은 차단된다.

    [시나리오 P1-S3] 입력 오류
    1. 사용자가 번호를 형식에 맞지 않게 입력한다.
    2. 시스템은 "올바른 핸드폰 번호를 입력해 주세요."를 노출한다.
    3. 진행은 차단되고, 사용자는 재입력한다.

## 3. P2 — 이름 아이디어 제출 페이지 (스크롤형 단일 페이지)

### 3-1. 화면 목적
세 공간(본당·소예배실·새가족부실)의 이름을 한 화면에서 입력받아 최종 제출한다.

### 3-2. 화면 구성 (스크롤 배치 순서)
    ① 본당 (조감도 이미지 + 설명)      → 이름1(필수) + 이름2(선택)
    ② 소예배실 (사진 + 설명)           → 이름1(필수) + 이름2(선택)
    ③ 새가족부실 (사진 + 설명)         → 이름1(필수) + 이름2(선택)
                                         → [제출하기]

> 미완공 상태인 **본당을 최상단**에 배치하여 조감도로 상상을 유도한다.

### 3-3. 공간별 구성 요소
| 공간 | 이미지 | 설명 문구(예시) | 입력 |
|------|--------|-----------------|------|
| 본당 | 조감도 | (설명 제공, 예시 이름 없음) | 필수1 + 선택1 |
| 소예배실 | 사진 | "청년과 청소년을 위한 액티브한 공간입니다." | 필수1 + 선택1 |
| 새가족부실 | 사진 | (설명 제공, 예시 이름 없음) | 필수1 + 선택1 |

### 3-4. 기능 정의
| 기능 ID | 기능명 | 설명 |
|---------|--------|------|
| F2-01 | 공간 정보 노출 | 이미지+설명 노출 (예시 이름 미제공) |
| F2-02 | 이름 입력 | 공간별 필수1 + 선택1 입력 |
| F2-03 | 필수값 검증 | 세 공간의 필수 이름 입력 여부 검증 |
| F2-04 | 최종 제출 | 신청자 정보 + 6개 필드 저장, 중복키 최종 확인 |
| F2-05 | 완료 이동 | 저장 성공 시 P3 이동 |

### 3-5. 비즈니스 로직 시나리오
    [시나리오 P2-S1] 정상 제출
    1. 사용자는 본당→소예배실→새가족부실 순으로 스크롤하며 확인한다.
    2. 각 공간별 이름1(필수)을 입력하고, 원하면 이름2(선택)도 입력한다.
    3. 사용자가 [제출하기]를 누른다.
    4. 시스템은 세 공간의 필수 이름 입력 여부를 검증한다.
    5. 시스템은 제출 직전 동일 번호의 중복 제출을 재확인한다.
       (동시 제출 등으로 인한 중복 방지)
    6. 이상이 없으면 신청자 정보 + 이름 데이터를 최종 저장한다.
    7. submitted_at(제출 일시)을 자동 기록한다.
    8. 시스템은 P3(제출 완료)로 이동한다.

    [시나리오 P2-S2] 필수 이름 누락
    1. 사용자가 일부 공간의 필수 이름을 비운 채 [제출하기]를 누른다.
    2. 시스템은 누락된 공간의 필수 필드에 경고를 노출한다.
    3. 제출은 차단되고, 사용자는 누락 항목을 입력 후 재시도한다.

    [시나리오 P2-S3] 제출 직전 중복 감지
    4. 사용자가 [제출하기]를 누른다.
    5. 시스템 재확인 결과 동일 번호가 이미 저장되어 있다.
    6. 시스템은 신규 저장을 차단하고 "이미 신청하신 분입니다."를 노출,
       [내용 확인하기](P4)로 유도한다.

    [시나리오 P2-S4] 저장 실패(서버 오류)
    7. 사용자가 [제출하기]를 누른다.
    8. 저장 처리 중 서버 오류가 발생한다.
    9. 시스템은 "일시적인 오류가 발생했습니다. 잠시 후 다시 시도해 주세요."를 노출한다.
    10. 입력값은 유지되어 재제출이 가능하다.

## 4. P3 — 제출 완료 페이지

### 4-1. 화면 목적
제출 완료를 알리고, 후속(2차 오프라인 스티커 투표) 일정을 안내한다.

### 4-2. 화면 구성 요소
| 구분 | 요소 | 내용 |
|------|------|------|
| 메시지 | 완료 안내 | "제출이 완료되었습니다. 참여해 주셔서 감사합니다!" |
| 안내 | 다음 일정 | "제안해 주신 이름 중 상위 후보는 추후 현장 스티커 투표로 최종 결정됩니다." |

### 4-3. 기능 정의
| 기능 ID | 기능명 | 설명 |
|---------|--------|------|
| F3-01 | 완료 메시지 노출 | 제출 성공 안내 |
| F3-02 | 후속 일정 안내 | 2차 투표 절차 안내 |
| F3-03 | 재진입 방지 | 완료 화면에서 재제출 경로 미노출 |

### 4-4. 비즈니스 로직 시나리오
    [시나리오 P3-S1] 완료 안내
    1. P2 저장 성공 시 시스템은 P3를 표시한다.
    2. 완료 메시지와 2차 투표 안내를 노출한다.
    3. 신규 제출을 위한 버튼은 제공하지 않는다. (제출 = 최종)
    4. 동일 번호로 재접근 시에는 P1에서 기존 신청자로 판별된다.

---

## 5. P4 — 내 제출 내용 조회 페이지 (읽기 전용)

### 5-1. 화면 목적
동일 번호 재접근 신청자에게 본인이 제출한 내용을 읽기 전용으로 보여준다.

### 5-2. 화면 구성 요소
| 구분 | 요소 | 내용 |
|------|------|------|
| 헤더 | 안내 문구 | "이미 신청하신 내용입니다." |
| 내역 | 신청자 정보 | 이름 |
| 내역 | 본당 | 이름1(필수), 이름2(선택) |
| 내역 | 소예배실 | 이름1(필수), 이름2(선택) |
| 내역 | 새가족부실 | 이름1(필수), 이름2(선택) |
| 안내 | 수정 불가 안내 | "제출 내용은 수정할 수 없습니다." |

### 5-3. 기능 정의
| 기능 ID | 기능명 | 설명 |
|---------|--------|------|
| F4-01 | 제출 내역 조회 | 번호 기준으로 저장 데이터 읽기 |
| F4-02 | 읽기 전용 표시 | 입력·수정 불가 상태로 표시 |
| F4-03 | (확장) 수정 진입 | ※ 정책 확정 시 추가 (현재 미구현) |

### 5-4. 비즈니스 로직 시나리오
    [시나리오 P4-S1] 제출 내역 조회
    1. P1에서 중복 감지된 사용자가 [내용 확인하기]를 누른다.
    2. 시스템은 해당 번호의 제출 데이터를 조회한다.
    3. 시스템은 신청자 이름 + 세 공간의 이름(필수/선택)을 읽기 전용으로 표시한다.
    4. 선택 이름(이름2)이 없으면 해당 항목은 미표시하거나 "-"로 표시한다.
    5. "제출 내용은 수정할 수 없습니다." 안내를 노출한다.

    [시나리오 P4-S2] 조회 데이터 없음(예외)
    6. 조회 시 해당 번호의 데이터가 조회되지 않는다.
    7. 시스템은 "조회할 내용이 없습니다."를 노출하고 P1(신청)로 유도한다.
